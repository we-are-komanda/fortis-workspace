# Продуктовый план Fortis / Конструктор защиты объекта

## Главная продуктовая цель

Сделать Fortis понятным конструктором защиты объекта, где пользователь может:

1. Создать или открыть объект.
2. Настроить эшелоны защиты.
3. Перетащить средства защиты на карту.
4. Увидеть размещённые объекты, покрытие и проблемные зоны.
5. Получить расчёт и отчёт по текущей конфигурации.
6. Показать заказчику понятную схему без технического мусора и лишней инженерной сложности.

---

# 1. Текущий фокус продукта

На ближайшем этапе **не разделяем средства на физические / нефизические / зональные**.

Для текущей версии считаем, что:

```txt
Все средства из библиотеки можно перетаскивать на карту как условные объекты эшелона.
```

Даже если в реальности OSINT, штаб или военная комендатура не являются физическими объектами на местности, в текущем прототипе они отображаются на карте как логические элементы защиты.

Главный сценарий:

```txt
Выбрал эшелон → выбрал средство → перетащил на карту → объект появился в эшелоне → расчёт обновился
```

# 2. Продуктовая версия 1 — понятный drag-to-map конструктор

## Цель

Сделать базовый сценарий размещения средств на карте качественным, понятным и визуально аккуратным.

## Что должно быть для пользователя

- Пользователь видит эшелоны на карте.
- Пользователь выбирает активный эшелон.
- Пользователь видит библиотеку средств.
- Пользователь перетаскивает средство на карту.
- После размещения появляется объект.
- Объект виден:
  - на карте;
  - в списке объектов эшелона;
  - в расчёте;
  - в отчёте.

## Что нужно довести

### 2.1. Карточки библиотеки

Карточки средств должны стать компактными и понятными.

В карточке должно быть только главное:

- название средства;
- категория / назначение;
- короткая характеристика покрытия;
- стоимость, если она включена;
- сколько уже размещено;
- визуальная подсказка, что карточку можно перетащить.

Убрать из карточек:

- большие картинки;
- перегруженные описания;
- крупный бейдж `РАЗМЕЩЕНО`;
- техническое слово `слот`;
- лишние кнопки.

### 2.2. Размещение на карте

При перетаскивании должно быть понятно:

- что пользователь сейчас тащит;
- куда можно положить объект;
- в какой эшелон он попадёт;
- что произойдёт после отпускания.

На карте нужно показывать:

- подсветку допустимой зоны;
- понятный marker после размещения;
- ошибку, если размещение невозможно.

### 2.3. Объекты на карте

Текущие белые мини-картинки нужно заменить на аккуратные tactical markers.

Маркер должен показывать:

- тип средства через пиктограмму;
- категорию через цвет;
- выбранное состояние;
- предупреждение или конфликт;
- количество, если нужно.

3D-картинки можно оставить в карточках и деталях, но не использовать как основной объект на карте.

## Критерий готовности версии

Пользователь без объяснений понимает:

```txt
Я беру средство из библиотеки, ставлю его на карту, и оно становится частью конфигурации.
```

# 3. Продуктовая версия 2 — управление размещёнными объектами

## Цель

Сделать размещённые объекты не просто маркерами, а полноценными элементами конфигурации.

## Что должно быть для пользователя

После размещения объекта пользователь может:

- выбрать объект на карте;
- увидеть объект в списке эшелона;
- открыть карточку объекта;
- удалить объект;
- изменить количество;
- посмотреть координаты;
- увидеть, к какому эшелону он относится;
- понять вклад объекта в расчёт.

## Что нужно реализовать на уровне UX

### 3.1. Панель объектов эшелона

В активном эшелоне должен быть список размещённых объектов.

Карточка объекта должна показывать:

- название;
- тип средства;
- эшелон;
- количество;
- статус;
- стоимость;
- наличие проблем;
- действия:
  - показать на карте;
  - редактировать;
  - удалить.

### 3.2. Связь карты и списка

Нужно двустороннее выделение:

- клик по объекту на карте выделяет его в списке;
- клик по объекту в списке выделяет marker на карте;
- удаление из списка убирает объект с карты;
- изменение объекта пересчитывает итог.

### 3.3. Состояния объекта

Минимальные состояния:

- обычный;
- выбранный;
- предупреждение;
- конфликт;
- неактивный / отключённый.

## Критерий готовности версии

Пользователь понимает, что каждый объект на карте — это управляемая сущность, а не просто декоративная иконка.

# 4. Продуктовая версия 3 — покрытие средств

## Цель

Показать не только “где стоит объект”, но и “что он закрывает”.

## Что должно быть для пользователя

При выборе объекта пользователь видит его зону покрытия.

Примеры:

- РЛС — круг или сектор;
- ОЭП — сектор обзора;
- РЭБ — зона подавления;
- инженерные средства — условная зона;
- организационные средства — условный marker без физического покрытия.

На этом этапе не нужна идеальная физическая точность. Важно заложить визуальную модель покрытия.

## Что нужно в интерфейсе

Добавить слой:

```txt
[x] Объекты
[x] Покрытие средств
[ ] Конфликты
[ ] Рельеф
```

Покрытие должно:

- включаться и выключаться;
- подсвечиваться для выбранного объекта;
- не превращать карту в шум;
- помогать увидеть непокрытые участки.

## Критерий готовности версии

Пользователь может ответить на вопрос:

```txt
Какие зоны уже прикрыты, а где есть пробелы?
```

# 5. Продуктовая версия 4 — конфликты и неправильное покрытие

## Цель

Сделать систему не просто редактором, а помощником, который показывает ошибки конфигурации.

## Какие проблемы нужно показывать

- объект размещён вне нужного эшелона;
- объект стоит в зоне, но покрытие не закрывает нужный сектор;
- между эшелонами есть разрыв;
- coverage объектов пересекается неэффективно;
- в эшелоне нет средств нужного типа;
- после изменения радиусов часть объектов стала конфликтной.

## Как это должно выглядеть

На карте:

- красный marker — ошибка;
- жёлтый marker — предупреждение;
- отдельный слой конфликтов;
- подсветка проблемной зоны.

В панели:

- список конфликтов;
- количество конфликтов по проекту;
- конфликты внутри карточки эшелона;
- конфликты внутри карточки объекта.

## Критерий готовности версии

Пользователь видит не только размещённые средства, но и понимает, где схема защиты слабая или некорректная.

# 6. Продуктовая версия 5 — рельеф / terrain layer

## Цель

Заложить отдельный слой рельефа и высот, который в будущем будет влиять на оценку покрытия.

## Что нужно сейчас

На первом этапе достаточно не полноценного расчёта, а продуктовой закладки:

- слой `Рельеф / высоты`;
- включение / выключение слоя;
- terrain warning у объекта;
- отображение terrain warnings в отчёте;
- подготовка UX под будущую линию видимости.

## Пользовательский смысл

Пользователь должен понимать:

```txt
Рельеф может повлиять на работу средств обнаружения и наблюдения.
```

## Критерий готовности версии

В интерфейсе есть место для рельефа, и он воспринимается как отдельный фактор оценки конфигурации.

# 7. Продуктовая версия 6 — переработка калькулятора

## Цель

Сделать калькулятор не отдельным способом собрать конфигурацию, а summary текущей карты.

## Что изменить

Калькулятор должен отвечать на вопросы:

- сколько объектов размещено;
- какие эшелоны заполнены;
- какие типы средств используются;
- где есть конфликты;
- какие зоны не покрыты;
- сколько стоит конфигурация, если стоимость включена.

## Что убрать из основного сценария

Убрать из главного UI:

- эталоны НАК / НЕВ / ФОСФОРИТ / БМУ;
- сравнение заводов по деньгам;
- ощущение, что конфигурация собирается кнопками `+ / −` в таблице.

## Что оставить вместо этого

Добавить:

- summary текущей карты;
- стоимость по эшелонам;
- количество объектов по эшелонам;
- количество объектов по категориям;
- конфликты;
- покрытие;
- переключатель `Показывать стоимость`.

## Критерий готовности версии

Калькулятор воспринимается как результат работы на карте, а не как отдельная независимая таблица.

---

# 8. Продуктовая версия 7 — новое сравнение конфигураций

## Цель

Убрать чувствительное сравнение заводов по стоимости и заменить его на безопасное инженерное сравнение.

## Что сравнивать

Не деньги как главный показатель, а структуру:

- количество эшелонов;
- количество объектов;
- количество средств обнаружения;
- количество средств подавления;
- количество средств поражения;
- количество средств пассивной защиты;
- количество конфликтов;
- количество непокрытых зон;
- terrain warnings;
- заполненность эшелонов.

## Возможные сценарии сравнения

- текущая конфигурация vs предыдущая версия;
- текущая конфигурация vs базовый шаблон;
- текущая конфигурация vs импортированная конфигурация;
- вариант А vs вариант Б.

## Критерий готовности версии

Пользователь может сравнить варианты защиты без раскрытия чувствительных данных и без ложной точности в деньгах.

# 9. Продуктовая версия 8 — отчёт для заказчика

## Цель

Сделать отчёт, который можно показать заказчику без дополнительных пояснений.

## Что должен содержать отчёт

- название объекта;
- дата формирования;
- карта эшелонов;
- список эшелонов;
- количество объектов;
- состав средств;
- зоны покрытия;
- конфликты;
- риски;
- рекомендации;
- стоимость, если включена;
- пометка, что расчёт предварительный / экспертный.

## Что важно

Если конфигурация пустая, отчёт не должен показывать старые бюджетные данные.

Если бюджетный подбор не применён, он не должен попадать в отчёт как факт.

## Критерий готовности версии

Отчёт можно отправить или показать заказчику как понятное summary текущей конфигурации.

# 12. Что НЕ делаем сейчас

На текущем этапе не выносим в приоритет:

- отдельную модель `map-object / non-physical / zone-object`;
- рисование сеток и периметров линией;
- отдельную логическую схему для OSINT/штабов;
- сложный физический расчёт рельефа;
- точную инженерную модель покрытия;
- полноценную backend-логику сохранения проектов.

Это можно оставить на следующие версии, когда базовый drag-to-map сценарий станет качественным.

# 14. Ключевой результат ближайшего этапа

После ближайших доработок пользователь должен получить понятный опыт:

```txt
Я открываю карту.
Вижу эшелоны.
Беру средство из библиотеки.
Перетаскиваю его на нужный эшелон.
На карте появляется аккуратный marker.
В панели появляется объект.
В расчёте обновляется конфигурация.
В отчёте появляется актуальное summary.
```

Если это работает чисто и красиво — продукт уже можно показывать как основу конструктора защиты.

## 1. Зачем это (концепт в одном абзаце)

Сегодня эксперты считают конфигурацию защиты **руками в Excel и рисуют слайды в PowerPoint**
(см. `docs/3CA74B5B-...pdf` — это и есть результат ручного труда). Каждый раз, когда меняется
бюджет, объект или набор средств, всё пересчитывается заново вручную. **Наш продукт
автоматизирует этот ручной расчёт**: ты выбираешь средства защиты и их количество по эшелонам,
а система мгновенно считает стоимость, приоритет, покрытие эшелонов и сравнивает варианты — то,
что в PDF сделано руками.

Формула из PDF (слайд 3): *«При масштабировании модели на новые активы изменяются только
переменные параметры. Логика расчёта при этом неизменна.»* — именно это мы и кодируем: один раз
зашиваем логику, дальше она работает на любых данных.

## 2. Что именно мы автоматизируем (что считаем)

Это ответ на твой вопрос «что мы можем считать». Пять блоков расчёта, от простого к сложному.
Каждый напрямую соответствует тому, что в PDF посчитано вручную.

### Блок A. Взвешенная оценка средства (заменяет колонку L в Excel)
**Вход:** оценки средства по 7 критериям (1–5 баллов) + веса критериев.
**Выход:** один балл 0–100.
**Что заменяет:** ручную формулу `=SUMPRODUCT(оценки; веса)` в Excel (колонка «Средневзв. оценка»).

Веса критериев берём из слайда 3 PDF (сумма = 100):

| Критерий | Вес |
|---|---:|
| Эффективность защиты / поражения | 25 |
| Скорость развёртывания | 25 |
| Стоимость | 20 |
| Масштабируемость | 10 |
| Доступность | 10 |
| Управляемость | 5 |
| Адаптивность | 5 |
| **Итого** | **100** |

> Замечание: в Excel-файле веса другие (Эффективность 25, Скорость 10, Стоимость 25, ...).
> **Берём веса со слайда 3 PDF как канонические** (это финальная экспертная версия), но веса
> делаем редактируемыми в UI — если эксперт захочет вернуть Excel-веса, он меняет числа, а не код.

### Блок B. Приоритет-цвет средства (заменяет ручную раскраску слайда 1)
**Вход:** взвешенный балл из блока A.
**Выход:** цвет приоритета — `зелёный` / `оранжевый` / `красный`.
**Что заменяет:** ручное решение эксперта «это красим зелёным, это красным» на слайде 1.

Правило (пороги выводим из PDF — там зелёный это высокий балл, красный низкий):
- `зелёный` (первоочередно): балл ≥ 70
- `оранжевый` (средний приоритет): 50 ≤ балл < 70
- `красный` (последний приоритет): балл < 50

Проверка по Excel: РЛС=79 → зелёный ✓, дроны-перехватчики=71 → зелёный ✓,
МОГ+турели=77 → зелёный ✓, воздушный патруль=58 → оранжевый, ствольная ПВО=57 → оранжевый,
лазер=35 → красный ✓. Совпадает с цветами PDF.

> Пороги тоже редактируемые в UI.

### Блок C. Смета конфигурации (заменяет ручное сложение на слайде 2) — **ГЛАВНОЕ**
**Вход:** список позиций `{средство, количество}`.
**Выход:** стоимость каждой позиции (`кол-во × цена`), итог по каждому эшелону, общий итог.
**Что заменяет:** ручной подсчёт `6 × 7 млн = 42 млн` и сложение колонок на слайде 2.

Это сердце калькулятора. Цены за единицу извлечены из PDF и **проверены арифметикой по всем
4 конфигурациям** (цены стабильны → модель верна):

| Средство | Цена за единицу |
|---|---:|
| Мобильный РЛС | 20 млн |
| Дроны-перехватчики (комплект) | 7 млн |
| Самолёты | 20 млн |
| Турельный комплекс | 7,9 млн |
| Бронеавтомобиль | 28 млн |
| Лазер | 690 млн |
| Ствольная ПВО 30 мм | 290 млн |

Формула:
```
СтоимостьПозиции = Количество × ЦенаЗаЕдиницу
ИтогЭшелона      = Σ СтоимостьПозиции по всем позициям эшелона
ИтогКонфигурации = Σ ИтогЭшелона по всем эшелонам
```

### Блок D. Покрытие эшелонов (заменяет ручную оценку «закрыт / не закрыт»)
**Вход:** конфигурация (какие эшелоны заполнены средствами).
**Выход:** для каждого из 8 эшелонов — закрыт он или пуст, и насколько (0–100%).
**Что заменяет:** ручную раскладку «в этой конфигурации эшелон 3 пустой» на слайде 2.

Простая, прозрачная модель (НЕ чёрный ящик):
```
ПокрытиеЭшелона = min(1, Σ (взвешенныйБалл средства / 100 × количество × вкладСредства) )
```
Где `вкладСредства` — нормирующий коэффициент, чтобы 1 РЛС не давала сразу 100%. На MVP можно
упростить до бинарного «есть средство в эшелоне → закрыт», а процент показывать как
`средний взвешенный балл средств эшелона`. **Это место мы обсудим — см. раздел 8 (открытые вопросы).**

### Блок E. Подбор под бюджет (то, чего в PDF нет — наша добавленная ценность)
**Вход:** бюджет (напр. 9,3 млрд) + каталог средств с ценами и приоритетами.
**Выход:** предложение «купи это первым» — жадный отбор по приоритету в рамках бюджета.
**Что заменяет:** интуицию эксперта «на что потратить деньги в первую очередь».

Логика (greedy by priority, простая и объяснимая):
```
1. Сортируй средства по приоритету: зелёные → оранжевые → красные,
   внутри цвета — по взвешенному баллу (выше = раньше).
2. Иди по списку, добавляй средство, пока хватает бюджета.
3. Покажи: что вошло, что не влезло, сколько денег осталось.
```
Это даёт ответ «на что потратить N млрд» с обоснованием через взвешенную оценку.

## 3. Какие данные нужны (модель данных)

```
КРИТЕРИЙ (criterion)
  id, название, вес (0–100)        ← 7 штук, сумма весов = 100

СРЕДСТВО ЗАЩИТЫ (asset)
  id, название, эшелонId
  ценаЗаЕдиницуRub                  ← из PDF (где есть) или экспертная заглушка
  единица: "шт" | "комплект"
  оценки: { эффективность, скорость, стоимость, масштабируемость,
            доступность, управляемость, адаптивность }   ← 1–5 баллов
  источникОценок: "excel" | "pdf" | "expert-stub"   ← честно помечаем происхождение

ЭШЕЛОН (echelon)
  id, номер (1..8), название, радиус (текст: "5–75 км")

ПОЗИЦИЯ КОНФИГУРАЦИИ (line item)
  средствоId, количество

КОНФИГУРАЦИЯ (configuration)
  id, название (НАК / НЕВ / ФОСФОРИТ / БМУ / своя), список позиций
```

Все оценки и цены лежат в одном файле данных (`infra/`), помечены источником. Заглушки честно
маркируются `expert-stub`, чтобы на экране можно было показать бейдж «предварительная оценка».

### Эшелоны (из PDF, 8 столбцов слайда 2)
1. Внешнее предупреждение — до 300 км
2. Обнаружение — 5–75 км
3. Подавление — 2–10 км
4. Огневое поражение (приоритет) — 1–20 км
5. Огневое поражение (резерв) — до 1,5 км
6. Пассивная защита / ИТЗ — 0,5–2 км, на узлах
7. АТЗ — n/a
8. Инфраструктура — на всех эшелонах

## 4. Экраны (UI) — концепт

> Финальный визуал прорабатываю отдельно через дизайн-скилл перед реализацией.
> Здесь — структура и логика экрана. Отдельный route `/calculator`, не трогает существующий
> `/prototype` (GIS/3D/риск).

### Экран 1. Каталог + конфигуратор (главный)
Таблица-матрица «эшелоны × средства», как слайд 2 PDF, но интерактивная:
- Слева — список эшелонов (1–8).
- В каждом эшелоне — карточки средств с: названием, ценой за единицу, **взвешенным баллом**
  (авто), **цветным бейджем приоритета** (авто), полем «количество».
- Меняешь количество → мгновенно пересчитывается стоимость позиции, итог эшелона, общий итог
  (блок C).
- Сверху — «Итого по конфигурации» большим числом + разбивка по эшелонам.
- Переключатель «загрузить эталон»: НАК / НЕВ / ФОСФОРИТ / БМУ → подставляет их количества.

### Экран 2. Сравнение конфигураций
Таблица: строки = эшелоны, колонки = НАК / НЕВ / ФОСФОРИТ / БМУ / «Моя».
Внизу — итоговые суммы. Подсветка где дороже/дешевле. Это слайд 2 PDF, но живой.

### Экран 3. Подбор под бюджет (блок E)
Поле «бюджет» → список «рекомендуем купить в таком порядке» с накопительной суммой и
отсечкой по бюджету. Объяснение каждой позиции: балл + приоритет.

### Панель «веса и пороги» (настройки модели)
Редактор 7 весов (с проверкой суммы = 100) и порогов цвета. Меняешь → пересчитываются все баллы
и цвета. Это делает модель «живой», а не захардкоженной.

## 5. Архитектура (по правилам проекта FSD)

Новый модуль `src/modules/defense-calculator/`:
```
defense-calculator/
├── domain/
│   ├── calculator-types.ts        # Criterion, Asset, Echelon, Configuration, ...
│   ├── scoring.ts                 # блок A (взвешенный балл) + блок B (цвет)
│   ├── costing.ts                 # блок C (смета) + блок D (покрытие)
│   ├── budget-fit.ts              # блок E (подбор под бюджет)
│   ├── scoring.test.ts            # проверка: РЛС=79, лазер=35, цвета
│   └── costing.test.ts            # проверка: НАК=2454.1млн и т.д.
├── infra/
│   └── catalog-data.ts            # каталог: цены (PDF), оценки (excel/stub), эталоны
└── ui/
    ├── calculator-page.tsx        # корневой компонент
    ├── echelon-matrix.tsx         # экран 1
    ├── comparison-table.tsx       # экран 2
    ├── budget-fit-panel.tsx       # экран 3
    └── weights-editor.tsx         # настройки модели
```
Route: `src/app/calculator/page.tsx` — тонкий, только рендерит `CalculatorPage`.

Слои импортируют строго вниз: `ui → domain → infra`. Расчёт (`domain`) чистый, без JSX и fetch —
поэтому легко тестируется и переносится на backend позже.

## 6. Как это работает — пошагово (для тебя)

1. Открываешь `/calculator`.
2. Видишь каталог: каждое средство уже с баллом и цветом (блок A+B посчитал автоматически).
3. Ставишь количества (или жмёшь «загрузить НАК») → видишь смету (блок C).
4. Жмёшь «Сравнить» → видишь 4 эталона рядом со своим вариантом (экран 2).
5. Вводишь бюджет → система предлагает порядок закупки (блок E, экран 3).
6. Если не согласен с оценкой средства — меняешь веса/баллы в настройках, всё пересчитывается.

Ничего из этого больше не считается руками — в этом вся суть.

## 7. Как проверять, что расчёт правильный (верификация)

Три уровня — чтобы ты сам мог убедиться, что цифры не выдуманы.

### Уровень 1. Автотесты (запускаются командой, я их напишу)
- `costing.test.ts`: загружает эталон НАК → ожидает итог 2 454,1 млн; НЕВ → 1 987 млн;
  ФОСФОРИТ → 3 446,6 млн; БМУ → 1 819,7 млн. Если расчёт сломается — тест покраснеет.
- `scoring.test.ts`: РЛС по своим оценкам → 79 (±погрешность); лазер → 35; проверка цветов.
- Запуск: `npm test` (или `npx vitest run`). Я покажу команду и её зелёный вывод.

### Уровень 2. Глазами в интерфейсе
- Загрузил НАК → вверху должно стоять ≈2,4 млрд. Сходится со слайдом 2? Значит верно.
- Цвета средств совпадают с цветами на слайде 1 PDF.

### Уровень 3. Ручная сверка одной позиции
- Дроны-перехватчики, количество 6 → должно показать 42 млн (6 × 7). Это та же арифметика,
  что эксперт делал руками.

> **Важно про достоверность:** цены и суммы — реальные, из PDF, и проверены. Оценки по 7
> критериям там, где их нет в Excel, — экспертные заглушки, **честно помеченные бейджем**. Мы не
> выдаём догадки за факты: модель правильная по структуре и по деньгам, а спорные баллы видно и
> можно поправить.

## 8. Решения (бывшие открытые вопросы)

1. **Блок D (покрытие эшелонов):** ✅ РЕШЕНО — процент через баллы. Покрытие эшелона = средний
   взвешенный балл средств эшелона (0–100%), плюс флаг «эшелон пуст». Это оценка, а не физика —
   так и подписываем.
2. **Эшелоны 1, 3, 7** в PDF часто без цен (n/a). Показываем их в смете как «организационные,
   без CAPEX» — они не ломают сумму (цена = 0).
3. **Аномалии PDF** (зафиксированы при извлечении): в НАК «1 182 млрд» — опечатка, читаем как
   1 182 млн; в ФОСФОРИТ АТЗ «20 млн» аномально мало (возможна опечатка). В данных оставляем
   значения, при которых суммы сходятся, и комментируем в коде.
4. **Единицы отображения:** числа в млн рублей внутри (целые/дробные), форматирование «2,4 млрд»
   на экране.
5. **Тесты:** плоские `.test.ts`-скрипты, бросающие исключение при ошибке, запускаются через
   `npx tsx <file>` — как уже принято в `drone-defense/domain/*.test.ts`.

## 9. Объём первой итерации (что делаю после согласования)

- Блоки A, B, C полностью + блок D в упрощённом виде + блок E базовый.
- Данные: все средства из PDF/Excel с ценами и оценками (заглушки помечены).
- Экран 1 (конфигуратор) + экран 2 (сравнение) + настройки весов. Экран 3 (бюджет) — базовый.
- Тесты на суммы 4 эталонов и на баллы.
- 3D/GIS/риск-движок не трогаю.
```
```

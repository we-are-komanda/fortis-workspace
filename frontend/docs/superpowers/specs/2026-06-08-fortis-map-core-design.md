## Секция 1. Модель данных и стор (фундамент)

Не вводим новую сущность размещения — расширяем существующий поток `Placement` + `upsertLocalPlacement`/`moveLocalPlacement`/removal.

**Изменения в `use-defense-studio-store.ts`:**

- Новое поле состояния: `selectedPlacementId: string | null` — единое состояние выбранного объекта (основа пункта 3).
- Новые экшены:
  - `selectPlacement(id: string | null)` — устанавливает `selectedPlacementId`.
  - `removePlacement(id: string)` — удаляет placement из `localPlacementsByScenario[scenarioId]`, сбрасывает выбор если удаляемый был выбран, вызывает пересчёт.
  - `placeAssetInSlot({ assetId? , catalogGroupId?, layerId, slotId })` — обёртка над `buildCatalogPlacement` + `upsertLocalPlacement`, размещение без обязательной кнопки.
- Гарантии целостности (пункт 1):
  - стабильный `id` для каждого placement;
  - запрет дубля одного типа средства в одном слоте — `placeAssetInSlot` возвращает отказ (для warning);
  - после add/remove счётчики пересчитываются через существующий `loadScenarioPack`.

**Граница:** всё в `domain/`, без JSX.

## Секция 2. Drag-drop из библиотеки в слот (пункты 1, 2)

Перевод размещения с «клик + кнопка» на **drag-to-map со snap в слот активного эшелона**. Кнопочный путь оставляем как фолбэк (тач/без мыши).

- **Источник перетаскивания** — карточки `assets-panel.tsx`: добавить `draggable`, `onDragStart` кладёт `assetId`/`catalogGroupId` в `dataTransfer`.
- **Зона приёма** — оверлей над deck.gl-канвасом в `gis-board.tsx`:
  - `onDragOver`: экранная точка → ближайший слот **активного** эшелона; подсветка целевого слота (preview).
  - `onDrop`:
    - над слотом активного эшелона → `placeAssetInSlot(...)` (snap);
    - вне любого эшелона/слота → отмена (битый объект не создаётся);
    - слот занят тем же типом → отмена + warning (дубль).
  - После drop стор пересчитывает счётчики (уже есть), маркер появляется сразу.
- **Чистый хелпер** `screenPointToSlot(point, activeLayer, slots): EchelonMapSlot | null` в `domain/` — тестируется контрактным тестом. `gis-board` только дергает его и рисует preview.

**Решение:** snap только к слотам **активного** эшелона (предсказуемо, совпадает с «drop внутри активного эшелона»).

## Секция 3. Список объектов эшелона + синхронизация выбора (пункты 2, 3)

**Новый компонент** `ui/echelon-objects-list.tsx` — инвентарь размещённых объектов активного эшелона.

- **Карточка объекта** (пункт 2): название, эшелон, количество, статус, стоимость. Данные — через `domain`-хелпер `describePlacement(placement): { name, echelon, qty, status, costMln }` (компонент в каталог не лезет).
- **Действия на карточке:**
  - выбрать → `selectPlacement(id)`;
  - показать на карте → центрировать `viewState` на `mapRef` (отдельная кнопка, не побочный эффект выбора, чтобы выбор не дёргал камеру);
  - удалить → `removePlacement(id)`.
- **Пустое состояние** (пункт 2): плейсхолдер «В этом эшелоне пока нет объектов».

**Синхронизация выбора (пункт 3)** через единый `selectedPlacementId`:
- клик по маркеру на карте → `selectPlacement(id)` → подсветка карточки;
- клик по карточке → `selectPlacement(id)` → подсветка маркера;
- удаление из списка → `removePlacement(id)` → маркер исчезает;
- selected визуально считывается везде (Секция 4).

## Секция 4. Состояния маркеров (пункт 4)

6 явных состояний для маркеров размещённых объектов. Триггеры — чистая функция `getMarkerState(placement, ctx): MarkerState` в `domain/` (тестируется). `gis-board` маппит состояние → цвет/радиус/обводку deck.gl.

| Состояние | Триггер | Визуал |
|---|---|---|
| **default** | размещён, всё в норме | цвет эшелона, обычный размер |
| **hover** | курсор над маркером | лёгкое увеличение + светлая обводка |
| **selected** | `selectedPlacementId === id` | тёмная толстая обводка + увеличенный размер |
| **warning** | низкий readiness объекта / эшелон ниже порога покрытия | жёлтая обводка |
| **conflict** | дубль типа в слоте/эшелоне или спор за слот | красная обводка + значок |
| **inactive** | объект выключен / readiness ≈ 0 | приглушённый, полупрозрачный |

- **Приоритет состояний:** `selected` > `conflict` > `warning` > `inactive` > `hover` > `default` (выбор всегда виден).
- Цвета согласованы с существующей палитрой `layerColors`; warning/conflict — жёлтый/красный поверх.
- Конкретные пороги (readiness, покрытие) вынесены константами в `domain/`, чтобы легко править.

## Секция 5. Слой покрытия (пункт 5)

Покрытие — **отдельный переключаемый слой**, НЕ встроенный в маркер.

- **Переключатель** «Показать покрытие» → `coverageVisible: boolean` в сторе (UI-состояние).
- **Что рисуем:** покрытие **выбранного** объекта (`selectedPlacementId`), чтобы не перегружать карту.
- **Типы покрытия:**
  - `circle` → `ScatterplotLayer` (полупрозрачная заливка по радиусу);
  - `sector` → `PolygonLayer`, полигон строится чистым хелпером `buildSectorPolygon(center, azimuth, angle, radius)` в `domain/`;
  - `zone` → существующая геометрия кольца эшелона;
  - `none` → ничего не рисуем.
- **Источник формы:** `domain`-хелпер `getCoverageShape(placement): CoverageShape` — берёт тип/радиус/угол из каталога; при отсутствии данных — разумные дефолты по типу средства (РЛС/ОЭП → сектор, огневое → круг и т.д.), в одном месте, легко править. `none` если неприменимо.

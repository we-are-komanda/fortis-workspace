# Единый источник эшелонов: costing на захардкоженных 8 эшелонах

- **Статус:** open
- **Обнаружено:** 2026-06-09 (контекст: пункты 6–8 калькулятора — эталоны/структурное сравнение/бюджет-флаг)
- **Область:** `defense-calculator` (домен costing + отчёт), частично пересекается с `shared` (слои проекта)
- **Серьёзность:** средняя

## Проблема

У эшелонов **два независимых источника правды**, и они не синхронизированы:

| | Источник | Кол-во | Динамичный |
|---|---|---|---|
| Карта / проект (`project.layers`) | [`src/shared/config/default-defense-layers.ts:11-20`](../../src/shared/config/default-defense-layers.ts#L11-L20) (L1…L9) через стор | 9 по умолчанию, до 20, можно создавать/удалять | да |
| Калькулятор (`echelons`) | [`src/modules/defense-calculator/infra/catalog-data.ts:35-42`](../../src/modules/defense-calculator/infra/catalog-data.ts#L35-L42) (ech_1…ech_8) | 8, фиксировано | нет |

Связь между ними — **односторонняя и приближённая**: маппинг кода слоя (L1…L9) → `echelonId` через `switch` в [`src/modules/defense-calculator/domain/project-asset-adapter.ts:20-43`](../../src/modules/defense-calculator/domain/project-asset-adapter.ts#L20-L43) (например, L4 и L7 оба сворачиваются в `ech_3_suppression`). Обратной синхронизации нет, и количества даже не совпадают (9 слоёв против 8 эшелонов).

Денежный costing считает по фиксированному списку: `estimateConfiguration` итерирует `context.echelons` (захардкоженные 8) — [`src/modules/defense-calculator/domain/costing.ts:65`](../../src/modules/defense-calculator/domain/costing.ts#L65). В калькуляторе этот список приходит из `catalog-data` — [`calculator-page.tsx:76`](../../src/modules/defense-calculator/ui/calculator-page.tsx#L76).

В печатном отчёте раздел **«1. Смета выбранной конфигурации»** строится из `myEstimate.echelons` — то есть из тех же захардкоженных 8 — [`calculator-report.tsx:118`](../../src/modules/defense-calculator/ui/calculator-report.tsx#L118).

## Почему так осталось

Цель пунктов 6–8 была: убрать эталоны (НАК/НЕВ/ФОСФОРИТ/БМУ) и денежное сравнение из ядра, перевести сравнение в структурное и починить бюджет-флаг. Эти поверхности уже читают живые слои проекта динамически:

- вкладка «Структура» (`buildStructuralProfile` → `calculateLayerSummaries(project)`),
- разделы отчёта «1.1. Эшелоны карты» и «2. Структурный профиль»,
- сводка «Смета по эшелонам карты» в Конфигураторе.

Унификация **самого costing-контура** (чтобы `estimateConfiguration` и раздел «1. Смета» тоже считали по `project.layers`, а захардкоженный список `echelons` ушёл из ядра) — это отдельный объём, который не входил в задачу. Поэтому осознанно оставили как долг, а не растягивали scope.

## Влияние

- Если пользователь добавит/удалит эшелон на карте, **структурные поверхности обновятся, а денежная смета по эшелонам — нет**: раздел «1. Смета» всегда показывает 8 строк из старого списка.
- Маппинг L→ech приближённый: разные слои сворачиваются в один calculator-эшелон, так что денежная разбивка по эшелонам не совпадает один-к-одному с реальной картой.
- Два списка названий/диапазонов эшелонов нужно поддерживать вручную; они уже разошлись (ср. «Срыв точности» L7 на карте vs его отсутствие как отдельного эшелона в калькуляторе).

## Как чинить

Сделать `project.layers` единственным источником правды для эшелонов и в costing:

1. Перевести `estimateConfiguration`/`CostingContext` на эшелоны, производные от `project.layers` (по `layerId`/`code`), вместо `context.echelons` из `catalog-data`.
2. Раздел отчёта «1. Смета выбранной конфигурации» строить по тем же живым слоям (как уже делают «1.1» и «2»).
3. Убрать захардкоженный `echelons` и приближённый `switch`-маппинг в `project-asset-adapter.ts` из ядра; если присвоение средства эшелону нужно — выводить его из реального `layerId` размещённого объекта, а не из `recommendedLayerCodes`.
4. Данные `echelons` в `catalog-data.ts` можно оставить на диске для совместимости тестов (`costing.test.ts`), но они не должны определять, какие эшелоны видит пользователь.

Это прямое продолжение пункта 6 («убрать legacy-данные, не связанные с текущей картой»).
